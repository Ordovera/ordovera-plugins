# MCP Server Governance Posture Audit

Analyze MCP server repositories for governance-relevant properties: who can do what, can you trace it, and what happens when it goes wrong. Combines automated static analysis (via `cc-mcp-audit`) with interpretive review.

This tool assesses governance posture, not vulnerability surface. It does not detect path traversal, SQL injection, prompt injection, or dependency CVEs. For deployment decisions, pair with SAST (`/top10-scan:sast-scan`) and dependency scanning (`/top10-scan:sca-scan`).

## Step 1: Check and Extract

Run the CLI against the user's input (URLs, local paths, or candidates file from `/mcp-audit:mcp-discover`):

```bash
npx cc-mcp-audit <source> --format json
```

For candidates files: `npx cc-mcp-audit -c <file> --format json`

For batches larger than 3 servers, write to file with `-o audit-raw.json`.

If `cc-mcp-audit` is not found, tell the user:

> `cc-mcp-audit` is not installed. From the ordovera-plugins repo, run:
> ```
> cd packages/cc-mcp-audit && npm install && npm run build
> ```
> Then re-run this skill.

## Step 2: Interpret

The CLI surfaces five named accountability gap patterns. For each server, review the automated findings and verify or refine them by reading source at the reported locations.

**ungated-write** (high confidence): Write tools with no confirmation gate co-located. Verify whether gates exist at the client layer or in middleware.

**global-auth-over-sensitive-tools** (high or low confidence): Uniform auth across tools with different sensitivity. Low confidence means the colocation heuristic could not determine auth architecture -- read the code. Verify whether the auth layer differentiates by operation.

**auth-without-actor-logging** (high confidence): Auth and logging both present, but log statements do not carry principal identifiers within proximity. Actions cannot be attributed to specific users.

**logging-without-attribution** (medium confidence): Logging present but no principal identifiers near log calls, and no auth detected. If deployed behind an external auth layer, log entries cannot tie actions to actors.

**destructive-without-audit-trail** (high confidence): Tools with irreversible operations (drop, delete, truncate, destroy) and no logging in their source file.

## Step 3: Present

Per server:

```
## [Server Name]
Source: [url] | Language: [lang]

### Tool Inventory
[Table: tool name, classification, sensitive keywords]

### Governance Posture
- **Authorization**: [Global / Per-tool / None / Delegated] -- [evidence]
- **Logging**: [Comprehensive / Partial / None] -- [attributed / unattributed]
- **Gates**: [Default-safe / Default-unsafe / Absent] -- [which tools are gated]

### Accountability Gaps
[Named gaps with instances and review notes from the CLI output]

### Risk Summary
Sensitive tools: [N of M] | Key concern: [one sentence]
```

For multi-server reports, include the comparison table (generated by `--format markdown`) and a patterns-across-servers summary.

## Notes

- Servers showing `hasAuth: false` may delegate auth to the MCP transport layer -- not necessarily a gap
- `hasAttributedLogging` requires principal identifiers within 3 lines of a log call in the same file -- file-wide presence of `user_id` does not count
- Low-confidence gaps (shown as `GA?` in the comparison table) require manual verification before citing
- "No tools extracted" with a framework-detected warning means the server uses a registration pattern not covered by regex extraction -- read source to complete the inventory
- When reading source at flagged locations, note tools accepting arbitrary input (SQL, shell commands, file paths) -- these are not accountability gaps but warrant mention in the risk summary
